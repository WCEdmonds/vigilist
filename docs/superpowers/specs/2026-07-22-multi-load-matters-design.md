# Multi-Load Matters — Add Loads to an Existing Matter

**Date:** 2026-07-22
**Depends on:** P0-SP5 source designation (merged), hotfix #44 (merged)
**Consumed by:** everyday ingest; P2-5 backlog resumes after this ships.

## Decision context (user direction, 2026-07-22)

The container (`Production` table) is functionally the MATTER — documents,
tags, redactions, privilege, QC, dedup, families, and production sets all
hang off it, and source designation assumes both directions of a case share
one container. But the ingest wizard creates a NEW container on every
upload, so users end up siloed per load. This feature makes the wizard able
to ADD a load (a received production or more of our own files) to the
matter it was opened in.

## Exploration findings that drive the design (verified 2026-07-22)

- Frontend uploads go browser → Firebase Storage directly
  (`uploadBytesResumable`, path built inline in `IngestWizard.tsx:158-160`)
  under the fixed prefix `productions/{id}/raw/{relativePath}`.
- `bootstrap_ingest_source` / `_download_dat_to_temp` pick the FIRST
  `.dat`/`.opt` under `raw/DATA/` — a second load's DAT would collide.
- `list_pdf_sources` / `list_native_sources` list-and-sort the whole
  `raw/` prefix; control numbers are `{prefix} {global_index+1:06d}` where
  `global_index` is the position in that sorted listing — adding files
  re-sorts the list, so a second load under the same prefix would produce
  colliding/shifted control numbers (existing docs keep stale numbers;
  new docs collide with them → silently skipped by the bates unique key).
- The wizard has no way to target an existing production; `App.tsx`
  mounts it with `{onClose, onComplete}` only.

## 1. Load namespace

Every NEW ingest (including the first load of a new matter) uploads under

```
productions/{id}/raw/loads/{load_id}/{relativePath}
```

with `load_id` = `crypto.randomUUID().slice(0, 8)` generated by the wizard.
Legacy files already at `raw/` root are untouched and never re-listed by a
namespaced load. Backend listing/bootstrap helpers gain an optional
`load_prefix` (e.g. `"loads/ab12cd34/"`):

- `bootstrap_ingest_source(production_id, load_prefix=None)` and
  `_download_dat_to_temp(...)`: DAT/OPT looked up under
  `raw/{load_prefix}DATA/`.
- `list_pdf_sources` / `list_native_sources(production_id, load_prefix=None)`:
  list `raw/{load_prefix}`.
- `load_prefix=None` preserves today's behavior exactly (legacy jobs,
  re-runs of old matters).

`load_prefix` rides in `IngestJob.field_mapping["load_prefix"]` (the
established custodian/source_party pattern); batch processors read it and
pass it to the listing/bootstrap helpers.

## 2. Control-number continuity across loads

For native/PDF loads, `/ingest/process` computes a **control offset** at
job start: the max numeric tail among existing `documents.bates_begin`
matching `^{derived prefix} \d+$` in the matter (0 when none). Stored as
`field_mapping["control_offset"]`; batch processors number
`{prefix} {offset + global_index + 1:06d}`. Second and later loads
continue the sequence instead of restarting at 000001 (which would collide
with the bates unique key and be silently skipped). Relativity loads keep
DAT-supplied Bates (cross-load duplicates correctly dedupe/skip).

## 3. Endpoints

- `/api/ingest/process` accepts `load_id` (sanitized: alnum, 1–32 chars →
  422 otherwise); folds `load_prefix` + `control_offset` into
  `field_mapping`; total-files counting (Cloud Tasks path) uses the
  namespaced listing.
- `/api/ingest/analyze` accepts `load_id`, threads the prefix to
  `analyze_load_file`.
- No new tables; no migration.

## 4. Wizard

- `IngestWizard` gains optional prop `existingProduction?: {id, name}`;
  App's in-matter mount passes the active matter (outside mount unchanged).
- When present, the setup stage leads with a target choice:
  **Add to {name}** (default) vs **Start a new matter** (shows the current
  name/description/context fields). Add-to skips `createProductionForIngest`,
  uses the existing id, generates `load_id`, uploads under the namespace,
  and passes `load_id` through analyze/process. The source-designation
  block (P0-SP5) is what distinguishes the load's direction.
- `onComplete` for add-to-matter refreshes the matter (existing
  `onIngestComplete` path).

## 5. Testing

- Backend: process endpoint folds `load_prefix`/`control_offset` (+422 on
  bad `load_id`); offset computation helper pure tests (empty, mixed
  prefixes, gaps); listing/bootstrap prefix construction with monkeypatched
  `list_files`/downloads; batch processors pass the prefix through
  (monkeypatched listers).
- Frontend: build gate.

## Out of scope

- UI "Matter" terminology rename (separate cosmetic sweep).
- Per-load management UI (list loads, delete a load) — later; the audit
  log and IngestJob rows already record each load.
- Re-numbering legacy matters' control numbers.
